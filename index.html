ヘルプ

このツールについて

これは、システム思考の「ループ図(因果ループ図)」を、複数人でリアルタイムに共同編集できるツールです。ノード(要因)と矢印(因果関係)を使って、物事のつながりを図にしていきます。

参加のしかた

  • 主催者から共有されたリンク(URL)を開いてください。
  • 閲覧だけであればログイン不要な場合があります。編集するにはログインが必要です。画面右上のメールアドレス欄に自分のメールアドレスを入力し「ログインリンクを送信」を押すと、届いたメール内のリンクをクリックすることでログインできます。
  • 画面右上のバッジで自分の権限を確認できます。「共同編集者」なら編集可、「閲覧のみ」なら見るだけです。

基本操作

  • 左のツールバーで「ノード」「矢印」「ループ」「テキスト」を選び、キャンバス上をクリックすると追加されます。
  • 要素をクリックすると選択され、右側にその内容(名前や色など)が表示されます。ドラッグで移動できます。
  • 矢印は、始点ノード→終点ノードの順にクリックして作成します。矢印を選択すると「+」(同方向)「-」(逆方向)の極性を設定できます。
  • 削除するには、要素を選択してから右側の削除ボタン、またはキーボードのDeleteキーを押します。
  • 何もない場所をドラッグすると画面全体を動かせます(パン)。マウスホイールやツールバーの+/-で拡大縮小できます。「全体表示」ボタンで図全体が画面に収まります。
  • キャンバス中央には図全体の「中心」を示す薄い十字マークが表示されます。図を大きく広げていくときの目印として使えます。

ループラベルの色

ループラベルの円の色は種類によって変わります。暖色(オレンジ系)は「強化ループ(R)」、寒色(青系)は「バランスループ(B)」を表します。

タブレットで使う場合

指1本でのドラッグ(移動・パン)に対応しています。ピンチでの拡大縮小は無効化されているため、拡大縮小はツールバーのボタンを使ってください。画面が縦だと右側のパネルが邪魔になりやすいため、できれば横画面での利用をおすすめします。

接続状態の見かた

画面下部の丸いマークで接続状態がわかります(緑=オンライン、黄=再接続中、赤=オフライン)。黄色や赤のときに表示される警告帯が出ている間は、自分の編集が他の参加者に届いていない可能性があるので、少し待ってから編集を続けてください。

その他

  • 同時に参加できる人数には上限があります。上限を超えると、後から参加した人が自動的に切断されることがあります。
  • 7日間まったく操作をしないと、自動的にログアウトされます。再度ログインリンクを受け取ってください。

主催者(オーナー)とは

図を最初に保存した人が「オーナー」になります。オーナーだけができる操作(共有設定の変更、サンプル読込、JSON読込、マイ図一覧の利用など)があります。

準備の流れ

  • ログインする(メールアドレスを入力し、届いたリンクをクリック)。
  • ノード・矢印・ループなどを使って図を作成する。
  • 「今すぐ保存」で保存する。保存すると図ごとのURLが発行されます(以降は自動保存もされます)。
  • 共有する前に、右上の共有設定を「リンクで閲覧可」または「リンクで編集可」に変更してください(初期値は「非公開」です)。
  • 「🔗 共有リンクをコピー」ボタンを押すと、URLが自動的にクリップボードにコピーされます。これをメールやチャットで参加者に送ってください。

共有設定の種類

  • 非公開：オーナー本人しか開けません。
  • リンクで閲覧可：リンクを知っている人は閲覧できますが、編集はできません。
  • リンクで編集可：リンクを知っていて、かつログインしている人は編集もできます(ログインしていない人は閲覧のみ)。

参加者の管理

  • 右上に接続中の参加者がバッジで一覧表示されます。
  • 同時接続には上限があります(オーナー以外は最大7人、オーナーを含めた合計で最大8人)。上限を超えて参加しようとした人は自動的に切断されます。
  • 個別に特定の参加者だけを退出させる機能は現状ありません。参加者を整理したい場合は、一度「共有設定」を「非公開」にしてから、再度「リンクで閲覧可/編集可」に戻し、参加してほしい人にだけ改めてリンクを送る、という運用にしてください(これにより古い参加者は接続が切れます)。

保存について

  • データベースへの自動保存は、オーナーの端末からのみ行われます(複数人が同時に保存して競合するのを防ぐためです)。他の参加者の変更も、いったんオーナーの端末に届いてから保存されます。
  • 「今すぐ保存」ボタンは、編集権限がある人なら誰でも押せます。
  • オーナー自身の接続が一時的に切れて復帰した場合、データベースの最新の保存内容で再同期されます。接続が切れていた間のオーナー自身のごく直近の未保存の編集(数秒分)は失われる可能性があります。
  • 画面を閉じようとしたときに未保存の変更が残っていると、ブラウザが確認メッセージを表示します。

マイ図一覧

ログイン後、「マイ図一覧」から自分が保存した図の一覧を確認できます。タイトルをクリックしてその場で名前を変更したり、「開く」で編集を再開したり、「削除」で完全に削除したりできます(削除は元に戻せません)。

書き出し(エクスポート)

  • JSONで書き出し：図データをファイルとして保存できます。バックアップや、別の図として複製したいときの読み込み元にも使えます。
  • 画像で書き出し：図全体をPNG画像として保存します。
  • PDFで保存：ブラウザの印刷ダイアログが開きます。そこから「PDFとして保存」を選んでください。

JSONの読み込みやサンプルの読み込み、マイ図一覧の利用は、共有中はオーナー以外は行えません(他の参加者の画面と食い違ってしまうためです)。図を差し替えたい場合はオーナーが行ってください。

AIにループ図のJSONを作ってもらう

組織の状況をAIとの対話から聞き取り、このツールに読み込めるJSONを作成させることもできます。以下のプロンプトをChatGPTやClaudeなどのAIチャットに最初のメッセージとして貼り付けて使ってください。対話の最後にJSONが出力されるので、それを.jsonファイルとして保存し、「JSONを読み込む」ボタンから読み込んでください。

あなたはシステム思考(ループ図・因果ループ図)の専門ファシリテーターです。目的は、ユーザーとの会話を通じて組織内の業務プロセスに潜む「悪循環」「好循環」の構造を明らかにし、その結果を後述のJSON形式で出力することです。

## 進め方

1. まずユーザーに、対象となる組織・チームの状況(業務内容、感じている問題、繰り返し起きていること、忙しさや疲弊感の背景など)を聞いてください。一度に全部聞き出そうとせず、対話形式で深掘りしてください。
2. 会話の中から、要因(例:「業務量」「ミスの発生」「残業時間」「モチベーション」など)を洗い出してください。要因は名詞または名詞句で、増減で捉えられるものにしてください(例:「疲労」であって「疲れている」ではない)。
3. 要因同士の因果関係を特定してください。それぞれの関係について、以下のルールで極性を判定してください。
   - +(同方向): 原因が増えると結果も増える/原因が減ると結果も減る
   - -(逆方向): 原因が増えると結果は減る/原因が減ると結果は増える
4. 矢印をたどって輪(ループ)になっている箇所を見つけたら、ループ内の「-」の数を数えてください。
   - 「-」が偶数個(0個を含む)→ 強化ループ(R): 変化がどんどん増幅される悪循環・好循環
   - 「-」が奇数個 → バランスループ(B): 変化を打ち消し合い安定化させる構造
5. ある程度要因・因果関係・ループが見えてきたら、ユーザーに一度整理内容を提示し、認識が合っているか確認してください。
6. 確認が取れたら、最後に以下の仕様に沿ったJSONを1つだけ、コードブロックで出力してください。JSONの前後に余計な説明は付けず、JSONのみを出力してください(そのままファイルに保存してツールに読み込ませるため)。

## 出力するJSONの仕様

このJSONは「ループ図エディタ」というWebツールにそのまま読み込ませるためのものです。以下の構造に厳密に従ってください。

```json
{
  "title": "図のタイトル(組織名やテーマ)",
  "nodes": [
    {
      "id": "node_1",
      "label": "要因名(短く)",
      "description": "補足説明(任意、空文字でも可)",
      "x": 0,
      "y": 0,
      "width": 160,
      "height": 60,
      "color": "#ffffff",
      "tags": []
    }
  ],
  "edges": [
    {
      "id": "edge_1",
      "sourceNodeId": "node_1",
      "targetNodeId": "node_2",
      "polarity": "+",
      "description": "この矢印が表す関係性の簡単な説明(任意)",
      "curve": 0
    }
  ],
  "loopLabels": [
    {
      "id": "loop_1",
      "label": "R1",
      "type": "reinforcing",
      "title": "ループの名前",
      "description": "このループがなぜ悪循環/好循環になるかの説明",
      "x": 0,
      "y": 0
    }
  ],
  "textNotes": []
}
```

### 各フィールドのルール

- `nodes[].id` / `edges[].id` / `loopLabels[].id` はすべて図全体で重複しない文字列にしてください(例: `node_1`, `node_2`, `edge_1` ...)。
- `edges[].sourceNodeId` と `targetNodeId` は、必ず `nodes` の中に存在する `id` を指定してください。
- `edges[].polarity` は `"+"` `"-"` `"unset"` のいずれかにしてください(不明な場合のみ `"unset"`)。
- `loopLabels[].type` は `"reinforcing"`(強化ループ) `"balancing"`(バランスループ) `"unclassified"`(未分類)のいずれかにしてください。
- `loopLabels[].label` は `R1`, `R2`, `B1`, `B2` のように種類ごとに連番を振ってください。
- `x`, `y` はノードの左上座標、`width` `height` はノードの大きさです。ノード同士が重ならないよう、1つのループを構成するノードは円状に(半径150〜220程度)配置し、ループごとに中心座標を500〜700程度離してください。`loopLabels` の `x`, `y` はそのループの中心付近に置いてください。
- `textNotes` は、複数のループが属する領域(部署名など)を示したい場合にのみ使ってください。不要なら空配列 `[]` のままで構いません。
- 単一の要因が複数のループに関わる場合、その要因のノードは1つだけ作成し、複数の矢印から共有してください(ノードを複製しない)。

## 注意点

- 実際の組織のヒアリング内容に基づかない、想像だけのループを作らないでください。情報が不足している場合は、JSONを出す前に質問して補ってください。
- 極性(+/-)とループの種類(強化/バランス)の判定は、上記のルールに従って必ず整合性を取ってください(自己チェックしてから出力してください)。
- ノードの数は多くなりすぎないよう、1つの会話につき目安10〜20要因程度に収めてください。多すぎる場合は、最も重要な2〜4個のループに絞ることを提案してください。

ワークショップ運営のコツ

  • 参加者にはあらかじめ「編集には自分のメールアドレスでのログインが必要」と伝えておくとスムーズです。
  • タブレットで参加する人がいる場合、横画面での利用を案内してください。
  • 接続状態が不安定な参加者がいる場合、画面上部に警告帯が出ていないか確認するよう伝えてください。

構成

本アプリは単一のHTMLファイル(index.html)にHTML/CSS/JavaScriptがすべて含まれており、ビルド不要です。GitHub Pagesでホストし、バックエンドにはSupabase(PostgreSQL + Auth + Realtime)を利用しています。

データ保存(diagramsテーブル)

図のデータは diagrams テーブルに保存されます。主なカラムは id(主キー)、title、data(ノード・矢印・ループ・テキスト注釈をまとめたJSONB)、owner_id(auth.usersのユーザーID)、share_settings(private / link-view / link-edit)、created_at / updated_at です。Row Level Security(RLS)により、オーナー本人は全操作可、share_settings が link-view / link-edit の場合は他ユーザーにも閲覧を許可、link-edit の場合はログイン済みユーザーの更新も許可、という設計にしています。詳細なポリシーはSupabaseダッシュボードのDatabase > Policiesで確認・編集できます。

認証

パスワードなしのマジックリンク(メールOTP)方式です。送信元メールアドレスはカスタムドメイン su-udon.com を取得し、Resend経由でSMTP認証を設定しています(Supabaseダッシュボードの Authentication > Emails / SMTP Settings)。

リアルタイム同期

  • 図ごとに diagram-{図のID} という名前のRealtimeチャンネルを使い、Broadcast({broadcast:{self:false}})で操作(ノード追加・移動・削除など)を配信しています。postgres_changes は意図的に使用していません。
  • 参加者の在席管理にはPresence機能を使っています。
  • Supabaseの制約として、Presenceのキー数は全プラン共通で1チャンネルあたり最大10という上限があります(Enterpriseプランのみ引き上げ交渉可)。これを踏まえ、アプリ側で同時参加人数のソフトな上限(オーナー以外7人、合計8人)を設け、上限超過者を自動切断する仕組みをクライアント側の決定的なロジック(参加時刻順のランク判定)で実装しています。サーバー側の強制ではないため、絶対的な保証ではありません。なお同時接続数(Free 200 / Pro 500)はプロジェクト全体で共有される制約であり、Presenceのキー数上限とは別軸で、図(チャンネル)ごとに独立しています。
  • 接続が切断→再接続した場合(初回接続時は除く)、データベースから最新状態を取得して再同期する処理を入れています(resyncFromServer)。

保存の一本化

自動保存(デバウンス付き)はオーナーのクライアントのみが実行するよう scheduleAutosave() 内でガードしています。複数クライアントが同時にデータベースへ書き込んで競合するリスクを避けるためです。手動の「今すぐ保存」ボタンはこのガードの対象外で、編集権限があれば誰でも実行できます(doSave() を直接呼び出すため)。

自動ログアウト

SupabaseのAuthセッション管理機能(タイムボックス化・非アクティブタイムアウト)はProプラン以上限定のため、本プロジェクトがFreeプランである前提でアプリ側(クライアントサイド)で疑似的に実装しています。localStorage に最終操作時刻(loopdiagram_lastActivity)を記録し、7日間以上更新がなければ次回アクセス時、および1時間ごとのチェックで supabase.auth.signOut() を呼び出します。JWT自体をサーバー側で強制失効させるものではなく、あくまでアプリ側の運用上の対策である点に注意してください。厳密なセッション管理が必要になった場合はProプランへのアップグレードを検討してください。

座標系と初期表示

図の座標系は原点(0,0)を中心とした自由な平面です。新規作成時やページ起動時は原点が画面中央に来るように初期表示されます(centerViewOnOrigin)。既存の図を開いたときは内容の範囲に合わせて自動的にフィットします(resetViewToDefault → fitToView)。キャンバス上の薄い十字マーク(#centerMark)は原点(0,0)を示す視覚的な目印で、実データではないため保存内容やPNG/PDF書き出しには含まれません。

デプロイ運用

コードの変更は、GitHubリポジトリの index.html を直接編集してコミットする運用です。GitHub Pagesが自動的に再デプロイします。ビルドツールやパッケージ管理は使用していません。

Supabase接続情報の変更

スクリプト冒頭付近にある SUPABASE_URL と SUPABASE_KEY(publishable key)を書き換えることで、接続先のSupabaseプロジェクトを変更できます。

既知の制約・注意点

  • 特定の参加者だけを個別に強制退出させる機能はありません(Supabaseダッシュボードからセッションを操作するか、共有設定を非公開→再公開する運用で代替してください)。
  • オーナー自身の接続が切断→再接続した際、直近の未保存編集(数秒分)がデータベースの内容で上書きされる可能性があります。
  • Supabase Freeプランのメール送信にはレート制限があります(Authentication > Rate Limits で確認可能)。短時間に大量のログインリンク送信を行うと制限に達することがあります。
